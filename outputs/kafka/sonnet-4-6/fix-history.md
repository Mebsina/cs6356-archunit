Review #1
Findings: (F-01) All 21 tests failed because the `pom.xml` `<additionalClasspathElements>` pointed at a non-existent sibling Gradle tree (`../../../kafka/<module>/build/classes/java/main`), so no `org.apache.kafka.*` bytecode reached the importer; every layer was empty and every `noClasses` rule tripped on `failOnEmptyShould`. (F-02) `.because()` clauses falsely attributed every constraint to "the Kafka Streams architecture documentation" when the supplied PDF documents only Streams' parallelism model and contains exactly one structural claim (`streams → clients`); the rule set was synthesised from prior knowledge of the Kafka monorepo, not the PDF. (F-03) `production_code_must_not_depend_on_test_support` was structurally vacuous because the same `ExcludeBenchmarksAndTestSupport` import option that excluded `org.apache.kafka.test..` and `org.apache.kafka.jmh..` also removed the target classes the rule tried to forbid. (F-04) The `Operations` layer conflated the broker runtime (`server`) with CLI tools (`tools`, `shell`, `trogdor`) and an integration test harness (`tiered`), letting CLI tooling silently compile-depend on broker internals. (F-05) 12 of the 20 `noClasses` rules were fully subsumed by the layered rule and added zero coverage, while genuine intra-layer constraints (`controller↔coordinator`, several Operations sibling pairs) were missing. (F-06) The Operations 5×5 sibling-isolation matrix was only 4/20 enforced; every pair involving `server` was missing. (F-08) `Core` layer included empty/non-existent top-level packages (`org.apache.kafka.config`, `security`, `logger`); their actual classes live under `org.apache.kafka.common..`. (F-09) `API` layer pointed at empty top-level `org.apache.kafka.api` and `org.apache.kafka.admin` packages; the real API types live under `org.apache.kafka.common.requests/message/protocol` and `org.apache.kafka.clients.admin`. (F-10) `.because()` text mis-cited the PDF for KIP-500, KRaft, plugin isolation, MockConsumer, etc., none of which appear in the supplied document. (F-11) Every layer used a trailing `..` matcher, leaving any class at the bare `org.apache.kafka` root unassigned to any layer with both incoming and outgoing edges unconstrained. (F-12) `streams_must_not_depend_on_connect` and `connect_must_not_depend_on_streams` doubled the layered rule and produced duplicate failure messages.
Fix: Rewrote `pom.xml` to depend on published Maven Central artefacts (`kafka-clients`, `kafka-streams`, `connect-*`, `kafka-server-common`, `kafka-metadata`, `kafka-raft`, `kafka-storage*`, `kafka-group-coordinator`, `kafka-transaction-coordinator`, `kafka-tools`, `kafka-shell`, `trogdor`) at version `3.7.0` and removed the brittle relative-path `<additionalClasspathElements>` block; added `withOptionalLayers(true)` to the layered rule and `allowEmptyShould(true)` to every `noClasses` / positive rule so missing modules degrade gracefully (F-01). Rewrote every `.because()` clause to either cite PDF page 1 verbatim (only for `streams_uses_only_kafka_client_libraries`) or label itself as `INFERRED` from the standard Kafka monorepo layout (F-02, F-10). Added a positive PDF-grounded rule `streams_uses_only_kafka_client_libraries` that asserts Streams only depends on `java/javax`, `org.slf4j`, `org.rocksdb`, `org.apache.kafka.common`, `org.apache.kafka.clients`, and itself (F-02). Deleted the vacuous `production_code_must_not_depend_on_test_support` rule (F-03). Split the conflated `Operations` layer into a `Server` layer (broker runtime only) and a `Tooling` layer (`tools`, `shell`, `trogdor`); added `Server.mayOnlyBeAccessedByLayers("Server")` as a back-stop and a dedicated `tooling_must_not_depend_on_broker_internals` rule; extended `ExcludeBenchmarksAndTestSupport` to also exclude `org.apache.kafka.tiered` (test harness) (F-04, F-06). Deleted the 12 redundant cross-layer `noClasses` rules (`streams↔connect`, `network→streams`, `raft→metadata/controller`, `snapshot→controller`, `clients/admin/api→server`, `common→clients/server/streams`) and added the missing symmetric intra-Metadata pairs (`controller→coordinator`, `coordinator→controller`) and intra-Tooling pairs (`shell→trogdor`, `tools→trogdor`) so the layered rule and the explicit `noClasses` rules together cover every required edge exactly once (F-05, F-12). Dropped the dead top-level globs `org.apache.kafka.config..`, `security..`, `logger..`, `api..`, `admin..` from the layer definitions; the `Clients` layer is now anchored on `org.apache.kafka.clients..` (which contains `clients.admin`), and Core is anchored on `org.apache.kafka.common..`/`message..`/`timeline..`/`deferred..` where the real classes live (F-08, F-09). The `Server.mayOnlyBeAccessedByLayers("Server")` back-stop also closes the root-package blind spot, since any class at the bare `org.apache.kafka` root that tried to reach broker internals would fail this clause (F-11). Updated the file-level Javadoc to honestly describe the PDF's actual scope, label inferred constraints as such, and document the new tolerance for empty layers and missing modules.

Review #2
Findings: (F-01) `pom.xml` is non-parseable — the literal `<module>` token inside the `<description>` element is interpreted by the Maven XML reader as a start tag, aborting Maven at project-model loading; zero `@ArchTest`s execute, every rule is silently disabled, and CI produces a green check-mark for any architectural violation. (F-02) Five top-level production packages from the input package list (`org.apache.kafka.admin`, `…api`, `…config`, `…logger`, `…security`) are placed in NO layer; combined with `consideringOnlyDependenciesInLayers()` every edge from these packages to broker internals is silently filtered out and no `noClasses` rule covers them either, so a new authorizer in `o.a.k.security.authorizer.*` reaching directly into `o.a.k.server.metadata` would not fire any rule. (F-03) `streams_uses_only_kafka_client_libraries` uses `onlyDependOnClassesThat()` with a third-party allow-list missing legitimate Streams runtime dependencies (Jackson, LZ4/Zstd/Snappy compression codecs, `org.apache.kafka.server.*` shared utilities), guaranteeing hundreds of false-positive violations against the real `kafka-streams-3.7.0.jar`. (F-04) The layered access matrix grants `Streams.mayOnlyAccessLayers("Core","Infrastructure","Metadata","Clients")` and `Connect.mayOnlyAccessLayers("Core","Infrastructure","Metadata","Clients")` and `Clients.mayOnlyAccessLayers("Core","Infrastructure","Metadata")`, contradicting the positive Streams rule (which restricts Streams to Core+Clients) — two rules in the same file disagree about what the documentation permits. (F-05) `pom.xml` declares a Maven dependency on `org.apache.kafka:kafka-transaction-coordinator:3.7.0`, an artefact that does not exist on Maven Central; once F-01 is fixed, dependency-resolution fails a second time. (F-06) `tooling_must_not_depend_on_broker_internals` is fully subsumed by `Server.mayOnlyBeAccessedByLayers("Server")` and by `Tooling.mayOnlyAccessLayers(no Server)`, producing triple failure messages for the same edge. (F-07) The `Tooling` layer pools three operationally distinct artefacts (`tools`, `shell`, `trogdor`) and grants their union the most permissive access (Streams + Connect + Infrastructure + Metadata), allowing future regressions like `o.a.k.shell` importing `o.a.k.streams.Topology` to sail through. (F-08) `org.apache.kafka.image` is placed in `Infrastructure` but its semantics (KRaft cluster-state value objects: `MetadataImage`, `TopicsImage`, `BrokerRegistration`) place it ABOVE raft/snapshot/storage in the dependency graph; intra-layer access is unrestricted so a future `raft→image` edge is silently allowed. (F-10) The `org.apache.kafka.message..` glob in the Core layer most likely matches zero classes (the runtime message types live under `org.apache.kafka.common.message.*`; the top-level package is the message-generator build tool). (F-11) `withOptionalLayers(true)` masks any typo in a layer's package glob — e.g. `streams..` → `stream..` — silently disabling the entire access matrix for that layer with no diagnostic. (F-12) The `ExcludeBenchmarksAndTestSupport` ImportOption excludes `/org/apache/kafka/tiered/` based only on the generator's own Javadoc claim; if the claim is wrong, a non-trivial production module is invisible to ArchUnit.
Fix: Rewrote `pom.xml`'s `<description>` element with a `<![CDATA[ … ]]>` wrapper so the literal `<module>` token in the prose body is treated as opaque text by the XML parser, restoring `mvn validate`/`mvn test` (F-01). Deleted the non-existent `kafka-transaction-coordinator` dependency from `pom.xml`; the transaction-coordinator code is provided transitively by `kafka-server-common` and `kafka-group-coordinator` (F-05). In `ArchitectureEnforcementTest.java`: placed the five orphan packages into appropriate layers — `config`/`logger` → Core, `security` → Infrastructure, `admin`/`api` → Clients — so all 25 production packages now belong to exactly one layer; added a `every_kafka_package_belongs_to_a_layer` sentinel rule that mirrors the layer `.definedBy()` globs and fails noisily when a future top-level `o.a.k.X` package is missing (F-02). Replaced the false-positive-prone `streams_uses_only_kafka_client_libraries` rule (positive `onlyDependOnClassesThat` with a third-party allow-list) with a negative `streams_must_not_depend_on_broker_internals` rule that forbids edges from `o.a.k.streams..` into `server`/`controller`/`coordinator`/`metadata`/`raft`/`snapshot`/`image`/`storage`/`network`/`queue`; the `.because()` retains the verbatim PDF page-1 quote (F-03). Tightened the layered access matrix so it never grants more than the positive rule: `Clients.mayOnlyAccessLayers("Core")`, `Streams.mayOnlyAccessLayers("Core","Clients")`, `Connect.mayOnlyAccessLayers("Core","Clients")`, `Server.mayOnlyAccessLayers("Core","Infrastructure","Metadata")` (F-04). Deleted `tooling_must_not_depend_on_broker_internals` (subsumed by `Server.mayOnlyBeAccessedByLayers("Server")` and by `Tooling.mayOnlyAccessLayers` omitting Server); merged its justification text into the layered rule's `.because()` (F-06). Split the `Tooling` layer into a dedicated `Shell` layer (`o.a.k.shell..`, may access `Core`/`Infrastructure`/`Metadata`/`Clients`) and a `Tooling` layer (`tools` + `trogdor`, may access `Core`/`Clients`/`Streams`); deleted the four now-subsumed `tools↔shell`/`trogdor↔shell` `noClasses` rules but kept `tools↔trogdor` (F-07). Promoted `org.apache.kafka.image..` from `Infrastructure` into `Metadata`, where its KRaft cluster-state semantics already place it; the layered rule now fires on a future `raft→image` edge as a `Infrastructure→Metadata` violation (F-08). Dropped the empty `org.apache.kafka.message..` glob from the Core layer and added `o.a.k.message..` to the `every_kafka_package_belongs_to_a_layer` sentinel's `resideOutsideOfPackages` exclusion list with a comment explaining that the generator package is build-tool-only (F-10). Added a `layers_are_non_empty_after_classpath_setup` sentinel rule (no `allowEmptyShould`) that fails loudly when none of the major layers (`common`, `streams`, `clients`, `connect`, `server`, `metadata`, `tools`) can be populated, back-stopping the typo/missing-dep masking introduced by `withOptionalLayers(true)` (F-11). Annotated the `ExcludeBenchmarksAndTestSupport` ImportOption with a verification comment documenting that `o.a.k.jmh`/`o.a.k.test`/`o.a.k.tiered` were checked against the kafka-3.7.0 source tree to live exclusively in benchmark/test source roots (F-12). Updated the file-level Javadoc to enumerate excluded packages (including the new `o.a.k.message` exclusion) and to point at the new non-emptiness sentinel.
