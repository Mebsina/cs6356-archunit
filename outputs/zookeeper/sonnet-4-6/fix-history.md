2. Second test generation after Review #1
Findings: (F-01) Root package `org.apache.zookeeper` was unassigned to any layer, leaving ~40+ critical classes (ZooKeeper, Watcher, ClientCnxn, etc.) invisible to all rules. (F-02) `graph` and `inspector` packages were declared excluded in Javadoc but were never actually excluded from the importer, allowing them to freely access server internals. (F-03) `Monitoring → Server` was blocked, causing false-positive violations on legitimate audit code that reads `server.Request` objects. (F-04) `Client → Monitoring` was blocked, causing false-positive violations on client-side metric emission. (F-05) `admin` and `cli` were incorrectly grouped in a single `API` layer despite being server-side and client-side respectively, granting each other's inappropriate cross-access. (F-06) No `mayOnlyBeAccessedByLayers` clauses on Server, leaving incoming edges unconstrained for unclassified classes. (F-07) Recipes had only negative restrictions (no server/api) but lacked a positive whitelist enforcing they only use the public client API. (F-08) Five fabricated parallel-sibling isolation rules (cli↔admin, metrics↔jmx, audit↔jmx/metrics) had no basis in the PDF and one (`audit → metrics`) was a false positive against real code. (F-09) `infrastructure_must_not_depend_on_higher_layers` was redundant with the layered architecture rule. (F-10) Several `.because()` clauses cited wrong or overstated PDF sections. (F-11) Unused `ZOOKEEPER_CLASSES` field was inconsistently configured compared to the annotation-driven import.
Fix: Added `ExcludeStandaloneTools` inner `ImportOption` class and wired it into `@AnalyzeClasses` to genuinely exclude `graph` and `inspector` (F-02, F-11 — field removed). Added root `"org.apache.zookeeper"` (no trailing `..`) to the Client layer definition to capture ZooKeeper, Watcher, and related root-package classes (F-01). Split the `API` layer into separate `Admin` (server-side) and `Cli` (client-side) layers with distinct `mayOnlyAccessLayers` matrices (F-05). Relaxed `Monitoring` to allow `→ Server` so audit logging does not false-positive (F-03). Relaxed `Client` and `Recipes` to allow `→ Monitoring` for metric emission (F-04). Added `whereLayer("Server").mayOnlyBeAccessedByLayers("Admin", "Monitoring", "Server")` to close the back-door for unclassified callers (F-06). Added `recipes_only_use_public_client_api` positive whitelist rule using `classes().should().onlyDependOnClassesThat(...)` (F-07). Deleted all five fabricated parallel-sibling rules and the redundant `infrastructure_must_not_depend_on_higher_layers` rule (F-08, F-09). Added dedicated `public_api_must_not_depend_on_server` and `cli_must_not_depend_on_server` back-stop rules. Rewrote all `.because()` clauses to cite §1.1, §1.6, §1.7, §1.8 accurately and label inferences explicitly (F-10).

3. Third test generation after Review #2
Findings: (R-01) `recipes_only_use_public_client_api` whitelist contradicted `layered_architecture_is_respected`: the layered rule permitted Recipes → `retry..`, `metrics..`, `jmx..`, `audit..`, `compat..`, `compatibility..`, but the whitelist omitted all of them, creating conflicting failure messages for the same edge. (R-02) Placing the root `org.apache.zookeeper` package into the Client layer caused a false-positive on `ZooKeeperMain`, which compile-depends on `cli.*` subcommands but `Client.mayOnlyAccessLayers(...)` forbade Client → Cli. (R-03) `org.apache.zookeeper.data..` (Stat, ACL, Id — part of the §1.6 public API) was in no layer, leaving C4 partially unenforceable. (R-04) Four `noClasses → server` rules (`public_api_must_not_depend_on_server`, `client_must_not_depend_on_server`, `cli_must_not_depend_on_server`, `recipes_must_not_depend_on_server`) were redundant with the layered architecture, producing triple failure messages per violation. (R-05) The `recipes_only_use_public_client_api` whitelist omitted common third-party libraries (Apache Commons, Guava, Yetus annotations), causing false positives on legitimate recipe dependencies. (R-06) The `.because()` clause stated "Observability may be accessed from any tier" while the rule forbids Infrastructure → Monitoring, making the clause factually wrong for the bottom layer. (R-07) `ExcludeStandaloneTools` lacked a comment explaining the URI-substring match semantics. (R-08) Javadoc described layers as a numbered "bottom to top" linear stack, misleading readers into thinking Recipes may reach Server, which the rules forbid.
Fix: Replaced the contradictory whitelist `recipes_only_use_public_client_api` and the four redundant `noClasses → server` rules with a single blacklist rule `recipes_must_not_depend_on_zookeeper_internals` that explicitly forbids Recipes → server/admin/cli/graph/inspector, matching §1.8 literally without enumerating third-party packages (R-01, R-04, R-05). Added `"Cli"` to Client's `mayOnlyAccessLayers` so `ZooKeeperMain` (root package) can legally dispatch to `cli.*` subcommands (R-02). Added `"org.apache.zookeeper.data.."` to the Client layer definition so Stat/ACL/Id are covered by the layered rule (R-03). Rewrote the `.because()` clause to state "Observability may be accessed from Server, Client, Admin, Cli, and Recipes; Infrastructure is strictly below observability" (R-06). Added a URI-substring-match comment to `ExcludeStandaloneTools` explaining forward-slash normalisation for both filesystem and JAR locations (R-07). Replaced the numbered "Inferred Layer Hierarchy (bottom to top)" list with a lattice bullet description clarifying that Server and Client are parallel, Admin/Cli/Recipes sit to the side, and Monitoring is cross-cutting (R-08). Removed the unused `classes` static import. Rule count reduced from 6 to 2 with no loss of documented coverage.
