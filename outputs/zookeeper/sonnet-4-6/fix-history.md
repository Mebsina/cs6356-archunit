2. Second test generation after Review #1
Findings: (F-01) Root package `org.apache.zookeeper` was unassigned to any layer, leaving ~40+ critical classes (ZooKeeper, Watcher, ClientCnxn, etc.) invisible to all rules. (F-02) `graph` and `inspector` packages were declared excluded in Javadoc but were never actually excluded from the importer, allowing them to freely access server internals. (F-03) `Monitoring → Server` was blocked, causing false-positive violations on legitimate audit code that reads `server.Request` objects. (F-04) `Client → Monitoring` was blocked, causing false-positive violations on client-side metric emission. (F-05) `admin` and `cli` were incorrectly grouped in a single `API` layer despite being server-side and client-side respectively, granting each other's inappropriate cross-access. (F-06) No `mayOnlyBeAccessedByLayers` clauses on Server, leaving incoming edges unconstrained for unclassified classes. (F-07) Recipes had only negative restrictions (no server/api) but lacked a positive whitelist enforcing they only use the public client API. (F-08) Five fabricated parallel-sibling isolation rules (cli↔admin, metrics↔jmx, audit↔jmx/metrics) had no basis in the PDF and one (`audit → metrics`) was a false positive against real code. (F-09) `infrastructure_must_not_depend_on_higher_layers` was redundant with the layered architecture rule. (F-10) Several `.because()` clauses cited wrong or overstated PDF sections. (F-11) Unused `ZOOKEEPER_CLASSES` field was inconsistently configured compared to the annotation-driven import.
Fix: Added `ExcludeStandaloneTools` inner `ImportOption` class and wired it into `@AnalyzeClasses` to genuinely exclude `graph` and `inspector` (F-02, F-11 — field removed). Added root `"org.apache.zookeeper"` (no trailing `..`) to the Client layer definition to capture ZooKeeper, Watcher, and related root-package classes (F-01). Split the `API` layer into separate `Admin` (server-side) and `Cli` (client-side) layers with distinct `mayOnlyAccessLayers` matrices (F-05). Relaxed `Monitoring` to allow `→ Server` so audit logging does not false-positive (F-03). Relaxed `Client` and `Recipes` to allow `→ Monitoring` for metric emission (F-04). Added `whereLayer("Server").mayOnlyBeAccessedByLayers("Admin", "Monitoring", "Server")` to close the back-door for unclassified callers (F-06). Added `recipes_only_use_public_client_api` positive whitelist rule using `classes().should().onlyDependOnClassesThat(...)` (F-07). Deleted all five fabricated parallel-sibling rules and the redundant `infrastructure_must_not_depend_on_higher_layers` rule (F-08, F-09). Added dedicated `public_api_must_not_depend_on_server` and `cli_must_not_depend_on_server` back-stop rules. Rewrote all `.because()` clauses to cite §1.1, §1.6, §1.7, §1.8 accurately and label inferences explicitly (F-10).

3. Third test generation after Review #2
Findings: (R-01) `recipes_only_use_public_client_api` whitelist contradicted `layered_architecture_is_respected`: the layered rule permitted Recipes → `retry..`, `metrics..`, `jmx..`, `audit..`, `compat..`, `compatibility..`, but the whitelist omitted all of them, creating conflicting failure messages for the same edge. (R-02) Placing the root `org.apache.zookeeper` package into the Client layer caused a false-positive on `ZooKeeperMain`, which compile-depends on `cli.*` subcommands but `Client.mayOnlyAccessLayers(...)` forbade Client → Cli. (R-03) `org.apache.zookeeper.data..` (Stat, ACL, Id — part of the §1.6 public API) was in no layer, leaving C4 partially unenforceable. (R-04) Four `noClasses → server` rules (`public_api_must_not_depend_on_server`, `client_must_not_depend_on_server`, `cli_must_not_depend_on_server`, `recipes_must_not_depend_on_server`) were redundant with the layered architecture, producing triple failure messages per violation. (R-05) The `recipes_only_use_public_client_api` whitelist omitted common third-party libraries (Apache Commons, Guava, Yetus annotations), causing false positives on legitimate recipe dependencies. (R-06) The `.because()` clause stated "Observability may be accessed from any tier" while the rule forbids Infrastructure → Monitoring, making the clause factually wrong for the bottom layer. (R-07) `ExcludeStandaloneTools` lacked a comment explaining the URI-substring match semantics. (R-08) Javadoc described layers as a numbered "bottom to top" linear stack, misleading readers into thinking Recipes may reach Server, which the rules forbid.
Fix: Replaced the contradictory whitelist `recipes_only_use_public_client_api` and the four redundant `noClasses → server` rules with a single blacklist rule `recipes_must_not_depend_on_zookeeper_internals` that explicitly forbids Recipes → server/admin/cli/graph/inspector, matching §1.8 literally without enumerating third-party packages (R-01, R-04, R-05). Added `"Cli"` to Client's `mayOnlyAccessLayers` so `ZooKeeperMain` (root package) can legally dispatch to `cli.*` subcommands (R-02). Added `"org.apache.zookeeper.data.."` to the Client layer definition so Stat/ACL/Id are covered by the layered rule (R-03). Rewrote the `.because()` clause to state "Observability may be accessed from Server, Client, Admin, Cli, and Recipes; Infrastructure is strictly below observability" (R-06). Added a URI-substring-match comment to `ExcludeStandaloneTools` explaining forward-slash normalisation for both filesystem and JAR locations (R-07). Replaced the numbered "Inferred Layer Hierarchy (bottom to top)" list with a lattice bullet description clarifying that Server and Client are parallel, Admin/Cli/Recipes sit to the side, and Monitoring is cross-cutting (R-08). Removed the unused `classes` static import. Rule count reduced from 6 to 2 with no loss of documented coverage.

4. Fourth test generation after Review #3
Findings: (R3-01) `consideringAllDependencies()` flagged every dependency to classes outside the defined layers — including `java.lang.Object`, `java.lang.Enum`, `javax.*`, `org.slf4j.*`, `org.apache.yetus.*`, `org.apache.jute.*`, `io.netty.*`, `org.jline.*` — producing 25,917 violations of which ≥99% were JDK/third-party noise that buried the real architectural findings. (R3-02) `ClientCnxn$EventThread` and `ClientCnxn$SendThread` (root / Client layer) extend `org.apache.zookeeper.server.ZooKeeperThread`, producing genuine Client → Server violations because `ZooKeeperThread` is a cross-cutting thread utility misplaced in `server..` by historical convention. (R3-03) `FileChangeWatcher$WatcherThread` (Infrastructure) extends the same `server.ZooKeeperThread`, producing a forbidden Infrastructure → Server edge for the same root cause. (R3-04) `org.apache.zookeeper.proto..`, `txn..`, and `version..` were in no layer and actively used; under `consideringOnlyDependenciesInAnyPackage`, unassigned ZK-internal targets still fire as violations. (R3-05) Root package assignment to Client layer sweeps in `ServerAdminClient` and other non-API utilities; `.because()` clause inaccurately described the root package as purely the §1.6 API surface. (R3-06) `ImportOption.DoNotIncludeJars` and `DoNotIncludeArchives` were absent, inflating scan scope with all third-party JARs on the classpath. (R3-07) `recipes_must_not_depend_on_zookeeper_internals` lacked a comment explaining that it is retained for §1.8 documentation readability despite being redundant with the layered rule.
Fix: Replaced `consideringAllDependencies()` with `consideringOnlyDependenciesInAnyPackage("org.apache.zookeeper..")` to scope enforcement to ZK-internal edges only, eliminating all JDK/third-party noise (R3-01). Added `ignoreDependency(simpleNameStartingWith("ClientCnxn").or(simpleName("FileChangeWatcher$WatcherThread")), simpleName("ZooKeeperThread"))` with an explicit "known architectural debt" comment to suppress the `ZooKeeperThread` false-positives until the class is moved upstream (R3-02, R3-03). Added `"org.apache.zookeeper.proto.."` and `"org.apache.zookeeper.txn.."` to the Client layer; added `"org.apache.zookeeper.version.."` to the Infrastructure layer, leaving no ZK-internal subpackage unassigned (R3-04). Updated the root-package comment and Javadoc lattice bullet to acknowledge that the Client layer contains both the §1.6 public API and historical utilities like `ServerAdminClient` (R3-05). Added `ImportOption.DoNotIncludeJars.class` and `ImportOption.DoNotIncludeArchives.class` to `@AnalyzeClasses` to limit scan to ZK's own compiled output (R3-06). Added a Javadoc comment on `recipes_must_not_depend_on_zookeeper_internals` explaining that the layered rule already covers the same constraint and this rule is intentionally redundant for §1.8 citation readability (R3-07). Added `import com.tngtech.archunit.core.domain.JavaClass` for the `ignoreDependency` predicates.

5. Fifth test generation after Review #4
Findings: (R4-01) `ignoreDependency` predicate was silently broken: `simpleNameStartingWith("ClientCnxn")` never matches `EventThread`/`SendThread` (ArchUnit simple name for a nested class is the nested segment only), and `simpleName("FileChangeWatcher$WatcherThread")` is a typo — the actual simple name is `WatcherThread`. All three suppressed violations continued to fire. (R4-02) `proto..`, `txn..`, and `data..` were classified as Client, but the server legitimately consumes every one of these shared wire-format records, causing ~1,100 Server → Client false-positive violations. (R4-03) Root package wholesale assigned to Client included Groups A/B — Watcher, KeeperException, ZooDefs, Op, Login, Environment, ZookeeperBanner, ZKUtil — all legitimately referenced by Server, producing ~300–400 more false positives. (R4-04) `Admin.mayOnlyAccessLayers(...)` omitted Client, but `ZooKeeperAdmin extends ZooKeeper` is the Admin class's entire design, generating ~40 violations. (R4-05) `Client.mayOnlyAccessLayers(...)` omitted Admin, but `ZooKeeperMain.connectToZK()` constructs `ZooKeeperAdmin`. (R4-06) `Monitoring.mayOnlyAccessLayers(...)` omitted Protocol, but `audit.AuditHelper` legitimately reads `proto.*` records and `CreateMode` to write audit log entries. (R4-07) Only `ZooKeeperThread` was attempted in the `ignoreDependency`; six other cross-cutting server.* utilities (`ByteBufferInputStream`, `ExitCode`, `ZooTrace`, `EphemeralType`, `auth.KerberosName`, `watch.PathParentIterator`) used outside the server tier were not addressed. (R4-08) Cli's `mayOnlyAccessLayers` omitted Admin, needed once ZooKeeperMain is reclassified to Cli. (R4-09) `.because()` clause referenced a non-existent Protocol layer and misstated Monitoring's actual permissions. (R4-11) Blacklist rule lacked a forward-compatibility note.
Fix: Introduced a dedicated `Protocol` layer (`data..`, `proto..`, `txn..` plus root-package API/cross-tier types matched by `ROOT_PROTOCOL_TYPES` predicate) sitting below both Client and Server, eliminating ~1,400 of 1,561 modelling false-positives (R4-02, R4-03). Added two static predicate constants — `ROOT_PROTOCOL_TYPES` and `ROOT_CLI_TYPES` — to split the root package by class rather than wildcard (R4-03, R4-05). Moved `ZooKeeperMain`, `JLineZNodeCompleter`, `Shell`, and `Version` to the Cli layer via `ROOT_CLI_TYPES` (R4-05). Rebuilt Client layer definition as an explicit predicate (root non-Protocol non-Cli classes + `client..` + `retry..`) removing `data..`/`proto..`/`txn..` (R4-02). Widened Admin to allow Client access (`ZooKeeperAdmin extends ZooKeeper`) (R4-04). Widened Cli to allow Admin access (`ZooKeeperMain.connectToZK()` uses `ZooKeeperAdmin`) (R4-05, R4-08). Added Protocol to every tier's `mayOnlyAccessLayers` list (R4-06). Rewrote `ignoreDependency` using `HasName.Predicates.name(...)` with fully-qualified class names and expanded to all seven known cross-cutting server.* utilities using `DescribedPredicate.alwaysTrue()` as origin (R4-01, R4-07). Rewrote `.because()` to accurately describe the Protocol layer and corrected Monitoring's documented permissions (R4-09). Added forward-compatibility note to the blacklist rule comment (R4-11). Added imports: `DescribedPredicate`, `or` (static), `name`/`nameMatching` (static from `HasName.Predicates`).


6. Sixth test generation after Review #5
Findings: (R5-01) ROOT_PROTOCOL_TYPES regex missed `OpResult(\\$.*)?`, `CreateOptions(\\$.*)?`, `Quotas`, `DeleteContainerRequest`, and the `SaslServerPrincipal(\\$.*)?` nested-class suffix — all public-API types legitimately referenced by the server, producing ~55 Server→Client false positives. (R5-02) `Shell` and `Version` were wrongly placed in ROOT_CLI_TYPES; `Shell` is a process-exec helper called by `Login$1.run()` and `Version` is a version-metadata provider called by four server classes, producing ~15 Server→Cli and Protocol→Cli false positives. (R5-03) ROOT_PROTOCOL_TYPES used `c.getPackageName()`/`c.getName()` directly; JVM array-type names (e.g. `[Lorg.apache.zookeeper.AddWatchMode;` for ENUM$VALUES fields) start with `[L` and do not match the regex, producing ~13 synthetic-enum violations per enum in the Protocol layer. (R5-04) `Infrastructure.mayOnlyAccessLayers("Infrastructure")` was too strict; `compat.ProtocolManager` has 34 edges to `proto.ConnectRequest/Response` and `util.SecurityUtils` constructs `SaslClientCallbackHandler` — both legitimate Infrastructure→Protocol edges, producing ~36 violations. (R5-05) `Client.mayOnlyAccessLayers(...)` omitted Admin; `client.ZooKeeperBuilder.buildAdmin()` explicitly constructs `admin.ZooKeeperAdmin` as a builder-pattern feature, producing 2 violations with no reclassification escape. (R5-07) Two more cross-cutting server.* utilities were missing from `ignoreDependency`: `server.auth.ProviderRegistry` (called by `common.X509Util`) and `server.util.VerifyingFileFactory` (called by `common.ZKConfig`), producing 12 violations. (R5-09) `ZKUtil` was wrongly in ROOT_PROTOCOL_TYPES; every public method takes a `ZooKeeper` argument, making it a Client-tier utility — placing it in Protocol produced 13 Protocol→Client violations. (R5-11) `.because()` clause still said Infrastructure "is the reusable, dependency-free base" after R5-04 widened it to allow Protocol access. (R5-08 — left as honest signal) `cli.GetConfigCommand` and `cli.ReconfigCommand` reach into `server.util.ConfigUtils` and `server.quorum.*` — genuine architectural smells intentionally kept visible.
Fix: Extended ROOT_PROTOCOL_TYPES regex with `OpResult(\\$.*)?`, `CreateOptions(\\$.*)?`, `Quotas`, `DeleteContainerRequest`, `SaslServerPrincipal(\\$.*)?`, `Shell(\\$.*)?`, `Version(\\$.*)?` (R5-01, R5-02, R5-06). Removed `ZKUtil(\\$.*)?` from the regex so it falls to the Client residual (R5-09). Added `rootPackageName(JavaClass)` and `rootName(JavaClass)` static helper methods that unwrap array types via `getBaseComponentType()`, and applied them in both ROOT_PROTOCOL_TYPES and the Client residual predicate to fix synthetic-enum classification (R5-03). Removed `Shell` and `Version` from ROOT_CLI_TYPES (R5-02). Widened `Infrastructure.mayOnlyAccessLayers` to include `"Protocol"` (R5-04). Added `"Admin"` to `Client.mayOnlyAccessLayers` (R5-05). Added `server.auth.ProviderRegistry` and `nameMatching("...VerifyingFileFactory(\\$.*)?")` to the `ignoreDependency` block (R5-07). Rewrote `.because()` to accurately reflect that Infrastructure consumes Protocol records (R5-11). Updated Javadoc Cli bullet to remove `Shell` and `Version`; updated Infrastructure description to mention Protocol access.
